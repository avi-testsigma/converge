# Workspace & Inbuilt Template Patterns

Patterns discovered from the Testsigma workspace inbuilt templates, test suites, and test data profiles. These represent production-grade patterns used internally.

---

## Reusability First

**Always prefer reusing existing templates and StepGroups over writing inline steps.**

### Priority Order
1. **Inbuilt templates** (`subtrees/inbuilt-templates/`) - ~170+ templates covering most common actions
2. **Custom StepGroups** - Create reusable BehaviorTrees for app-specific flows (login, navigation, etc.)
3. **Element registry** - Define locators once in `elements.sigma`, reference everywhere
4. **Inline steps** - Only for truly unique, one-off operations

### Using Inbuilt Templates

```xml
<!-- Instead of writing inline click logic -->
<Locate css="#submit" as="_btn" />
<Click element="_btn" />

<!-- Use the inbuilt template -->
<StepGroup ID="template-click" element="submit-btn" />

<!-- Instead of writing inline text entry -->
<Locate css="#email" as="_input" />
<Fill element="_input" text="admin@example.com" />

<!-- Use the inbuilt template -->
<StepGroup ID="template-enter-text" element="email-input" text="admin@example.com" />
```

## Element Registry (elements.sigma)

Elements are defined in a central registry file and referenced by ID via `LocateElement`:

```xml
<!-- elements.sigma - Element definitions -->
<root>
  <BehaviorTree ID="login-email-input">
    <Locate css="input[name='email']" as="_el" />
  </BehaviorTree>

  <BehaviorTree ID="login-password-input">
    <Locate css="input[type='password']" as="_el" />
  </BehaviorTree>

  <BehaviorTree ID="login-submit-btn">
    <Locate xpath="//button[@type='submit']" as="_el" />
  </BehaviorTree>
</root>
```

Then in test files:
```xml
<!-- Uses element registry to resolve the locator -->
<LocateElement ID="login-email-input" as="_el" />
<Fill element="_el" text="admin@example.com" />
```

### LocateElement vs Locate
| Feature | `LocateElement` | `Locate` |
|---------|----------------|----------|
| Source | Element registry (elements.sigma) | Inline definition |
| Reusability | High - defined once, used many times | Per-use |
| Maintenance | Change locator once, affects all tests | Change in each test |
| Best for | Large test suites with shared elements | Quick tests, unique elements |

### Element Naming Conventions
- **Human-readable**: `login-email-input`, `dashboard-nav-link` (preferred for hand-written tests)
- **Numeric IDs**: `element-742`, `element-1508` (auto-generated by Testsigma platform)

## Test Data Profiles

JSON-based parameterized data for data-driven tests:

```json
{
  "testDataProfileName": "LoginCredentials",
  "isMigrated": true,
  "testDataSets": [
    {
      "name": "valid-admin",
      "data": {
        "username": "admin@example.com",
        "password": "secret123",
        "expectedRole": "Admin"
      }
    },
    {
      "name": "valid-user",
      "data": {
        "username": "user@example.com",
        "password": "userpass",
        "expectedRole": "Member"
      }
    }
  ]
}
```

### Using Test Data in Tests

```xml
<!-- Interpolation syntax -->
<Fill element="_emailInput" text="${testData['username']}" />
<Fill element="_passInput" text="${testData['password']}" />
<ExpectText element="_roleLabel" expected="${testData['expectedRole']}" />
```

### ForEach with Test Data Profiles

```xml
<!-- Data-driven iteration over all data sets in a profile -->
<ForEach collectionProfileId="LoginCredentials" item="currentSet" index="i">
  <Sequence>
    <Fill element="_emailInput" text="${testData['username']}" />
    <Fill element="_passInput" text="${testData['password']}" />
    <Click element="_submitBtn" />
    <ExpectText element="_roleLabel" expected="${testData['expectedRole']}" />
    <LogMessage level="info" message="Verified data set ${i}" />
  </Sequence>
</ForEach>
```

## Script vs EvaluateScript

Two different JavaScript execution contexts:

### Script (BT Runtime)
Runs in the Node.js behavior tree runtime. Has access to the **blackboard** (test variables).

```xml
<!-- Compute a value and store it on the blackboard -->
<Script>
  <![CDATA[
    const price = parseFloat(blackboard.get('priceText').replace('$', ''));
    const tax = price * 0.08;
    blackboard.set('totalWithTax', (price + tax).toFixed(2));
  ]]>
</Script>
<AssertValue actual="${totalWithTax}" operator="equals" expected="108.00" />
```

### EvaluateScript (Browser Context)
Runs JavaScript **inside the browser page** via `page.evaluate()`. Has access to the DOM but NOT the blackboard.

```xml
<!-- Execute JS in the browser -->
<EvaluateScript outputKey="scrollHeight">
  <![CDATA[
    return document.body.scrollHeight;
  ]]>
</EvaluateScript>

<!-- With element context -->
<EvaluateScript element="_myElement" outputKey="computedColor">
  <![CDATA[
    return window.getComputedStyle(arguments[0]).color;
  ]]>
</EvaluateScript>
```

### Key Differences

| Feature | `Script` | `EvaluateScript` |
|---------|----------|------------------|
| Runs in | Node.js (BT runtime) | Browser (page context) |
| Access to | Blackboard, test variables | DOM, window, document |
| Use for | Data manipulation, assertions | DOM queries, browser state |
| CDATA needed | Yes (for special chars) | Yes (for special chars) |

### CDATA Sections
Always wrap JavaScript in `<![CDATA[...]]>` to avoid XML parsing issues with `<`, `>`, `&`:

```xml
<!-- WRONG: XML parser breaks on < and & -->
<Script>if (x < 10 && y > 5) { ... }</Script>

<!-- CORRECT: CDATA protects special characters -->
<Script><![CDATA[if (x < 10 && y > 5) { ... }]]></Script>
```

## Proxy Template Routing

A single template handles multiple behaviors based on a test data parameter:

```xml
<BehaviorTree ID="template-keyboard-action" params="element, testData, key">
  <Sequence>
    <LocateElement ID="${element}" as="_el" />
    <Conditional>
      <Conditional.If>
        <Script><![CDATA[
          blackboard.set('_isEnter', blackboard.get('testData') === 'Enter');
        ]]></Script>
        <AssertValue actual="${_isEnter}" operator="equals" expected="true" />
      </Conditional.If>
      <Conditional.Then>
        <Press element="_el" key="Enter" />
      </Conditional.Then>
      <Conditional.ElseIf>
        <Script><![CDATA[
          blackboard.set('_isTab', blackboard.get('testData') === 'Tab');
        ]]></Script>
        <AssertValue actual="${_isTab}" operator="equals" expected="true" />
      </Conditional.ElseIf>
      <Conditional.ElseIf.Then>
        <Press element="_el" key="Tab" />
      </Conditional.ElseIf.Then>
      <Conditional.Else>
        <Press element="_el" key="${key}" />
      </Conditional.Else>
    </Conditional>
  </Sequence>
</BehaviorTree>
```

## Table Navigation with Nested Within

For interacting with specific cells in tables:

```xml
<!-- Navigate: table → row → cell -->
<LocateElement ID="data-table" as="_table" />
<Within element="_table">
  <Locate role="row" as="_rows" />
  <Nth elements="_rows" index="${rowIndex}" indexBase="1" as="_targetRow" />
  <Within element="_targetRow">
    <Locate role="cell" as="_cells" />
    <Nth elements="_cells" index="${colIndex}" indexBase="1" as="_targetCell" />
    <Click element="_targetCell" />
  </Within>
</Within>
```

## ForceSuccess for Optional Steps

Wrap non-critical steps so they don't fail the test:

```xml
<!-- Optional cookie banner dismissal -->
<ForceSuccess>
  <Sequence>
    <Locate text="Accept Cookies" as="_cookieBtn" timeout="2000" />
    <Click element="_cookieBtn" />
  </Sequence>
</ForceSuccess>
<!-- Test continues regardless -->
```

### Production Template Pattern (with Timeout + Screenshot)

The inbuilt templates wrap steps with ForceSuccess, Timeout, and Screenshot for robustness:

```xml
<BehaviorTree ID="template-click-robust" params="element">
  <ForceSuccess>
    <Timeout msec="30000">
      <Sequence>
        <LocateElement ID="${element}" as="_el" />
        <Click element="_el" />
        <Screenshot path="./screenshots/step-evidence.png" />
      </Sequence>
    </Timeout>
  </ForceSuccess>
</BehaviorTree>
```

## Invert for Negative Assertions

Use `<Invert>` to assert the opposite of a positive check:

```xml
<!-- Verify element is NOT present -->
<Invert>
  <Locate text="Error" as="_error" timeout="2000" />
</Invert>

<!-- Verify alert did NOT appear -->
<Invert>
  <HandleDialog action="accept" timeout="2000" />
</Invert>
```

## RetryUntilSuccessful for Flaky Operations

```xml
<!-- Retry clicking a dynamically loaded button -->
<RetryUntilSuccessful num_attempts="5" delay_ms="1000">
  <Sequence>
    <Locate css=".async-button" as="_btn" timeout="3000" />
    <Click element="_btn" />
    <ExpectVisible element="_successMsg" expectedState="true" timeout="2000" />
  </Sequence>
</RetryUntilSuccessful>
```

## Parallel for Event Listening

Wait for a new page/tab while triggering the action:

```xml
<!-- Must start listening BEFORE the click triggers the event -->
<Parallel strategy="strict">
  <WaitForNewPage as="_newPage" timeout="10000" />
  <Click element="_linkThatOpensNewTab" />
</Parallel>
<SwitchToPage pageRef="_newPage" />
```

## Two Format Generations

The workspace shows two distinct template patterns:

### Modern/Lightweight Format
Direct, minimal wrapping:
```xml
<BehaviorTree ID="template-click" params="element">
  <Sequence>
    <LocateElement ID="${element}" as="_el" />
    <Click element="_el" />
  </Sequence>
</BehaviorTree>
```

### Legacy/Full Format
Wrapped with ForceSuccess, Timeout, and Screenshot:
```xml
<BehaviorTree ID="template-click" params="element">
  <ForceSuccess>
    <Timeout msec="30000">
      <Sequence>
        <LocateElement ID="${element}" as="_el" />
        <Click element="_el" />
        <Screenshot path="./evidence.png" />
      </Sequence>
    </Timeout>
  </ForceSuccess>
</BehaviorTree>
```

**Recommendation**: Use the lightweight format for hand-written tests. The full format is auto-generated by the Testsigma platform for production test runs where evidence collection and graceful failure are needed.

| Selenium Key | Playwright Key |
|-------------|---------------|
| `ENTER` / `RETURN` | `Enter` |
| `TAB` | `Tab` |
| `ESCAPE` | `Escape` |
| `BACK_SPACE` | `Backspace` |
| `DELETE` | `Delete` |
| `SPACE` | `Space` |

| `ARROW_DOWN` | `ArrowDown` |
| `ARROW_LEFT` | `ArrowLeft` |
| `ARROW_RIGHT` | `ArrowRight` |
| `CONTROL` | `Control` |
| `META` | `Meta` |
| `ALT` | `Alt` |
| `SHIFT` | `Shift` |

```xml

<Press element="_input" key="Control+a" />

```

## Template Categories (Inbuilt)

The ~170+ inbuilt templates cover these categories:

1. **Click Actions**: click, double-click, right-click, click-with-alert, click-new-window
2. **Text Entry**: fill, clear-and-fill, fill-with-enter, type (character-by-character)
3. **Keyboard Actions**: press key, key combos, Selenium-to-Playwright mapping
4. **Dropdown/Select**: select by value, label, index
5. **Checkbox/Radio**: check, uncheck, select radio
6. **Verification**: visible, not-visible, text-contains, text-exact, value, enabled, disabled, count
7. **Navigation**: navigate, go-back, go-forward, reload, wait-for-url
8. **Data Extraction**: get-text, get-attribute, get-value, get-title, get-url
9. **Scroll Actions**: scroll-to-element, scroll-to-position, scroll-within-element
10. **Wait Actions**: wait-for-element, wait-for-text, wait-for-url, wait-for-load
11. **Page Management**: new-tab, switch-tab, close-tab, get-page-count
12. **Advanced**: drag-and-drop, file-upload, iframe, dialog, cookie management
